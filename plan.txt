Context
The Umbraco.AI.Agent package currently manages individual AI agents. Users need a way to visually compose multiple agents into workflows using Microsoft Agent Framework (MAF) orchestration patterns. This feature adds a visual workflow builder within the existing Agent package, allowing users to create, edit, and execute multi-agent orchestrations directly from the Umbraco backoffice.
Key decisions made:

Orchestrations live within Umbraco.AI.Agent (not a separate product)
All five MAF patterns supported from day one: Sequential, Concurrent, Handoff, Group Chat, Magentic
Rete.js with @retejs/lit-plugin for the visual editor (official Lit support, framework-agnostic)


1. Design Answers
Q1: Should orchestrations be exposable as Agents with surface configuration?
Yes. MAF explicitly supports "workflows as agents" — an orchestration can be used anywhere a single agent can. The implementation:

AIOrchestration gets its own SurfaceIds property (same pattern as AIAgent)
When assigned to a surface (e.g., "copilot"), the orchestration appears alongside regular agents
At runtime, the system creates a MAF workflow and wraps it as an agent via MAF's built-in composability
Users can chat with an orchestration the same way they chat with a single agent

Q2: Do we only orchestrate agents from the agents section?
Primarily yes, but with one addition:

Agent references: Nodes reference existing AIAgent entities by ID — the primary mechanism
Function nodes: These are registered M.E.AI AITool instances that execute without an LLM — data transformation, API calls, aggregation logic. In MAF terms these map to "Function" executors. The UI presents the same tool picker already used in the agent editor
No inline agents: Creating full agents should happen in the agents section, not embedded in workflows. This keeps the agent as the single source of truth

Q3: Is there a need for prompts within the workflow?
Not at the orchestration level. MAF orchestrations coordinate agents — they don't have their own prompts. Each agent node uses the referenced agent's instructions (which may already use Umbraco.AI.Prompt templates). The orchestration layer deals with routing and coordination, not prompt engineering.
Specific node types and prompts:

Agent nodes: Use the referenced agent's instructions as-is (no override in v1)
Router nodes: Use structured rule-based conditions (see section on conditions below), not prompts
Aggregator nodes: Use a strategy selection ("concat", "vote", "summarize") — the "summarize" strategy internally uses an LLM via a profile, but this is configuration, not a user-authored prompt
Manager nodes (Magentic): Have lightweight coordination instructions that tell the manager how to delegate — this is the one place where free-text instructions exist at the orchestration level

Q4: Is an orchestration a single MAF orchestration, or can patterns be mixed?
The graph is freeform. There is no Pattern field on the entity — the pattern is purely a UI concept for the creation experience. When creating a new orchestration, a modal offers "Blank" or one of the five pattern templates as starting points. Once the graph is populated, the pattern is implicit in the graph structure. A user could start from a Sequential template and add a concurrent branch. Each orchestration maps to exactly one MAF Workflow at execution time.
Q5: Why Rete.js over AntV X6?
AntV X6 is React-focused — its custom node rendering (@antv/x6-react-shape), component library (@antv/x6-react-components), and documentation all center on React. The Umbraco.AI frontend uses Lit web components. Rete.js has an official @retejs/lit-plugin with first-class Lit support — custom nodes are rendered using html tagged templates and customElements.define(). No framework bridging or React dependency needed.

Q6: What are condition expressions?
Router node conditions use a structured rule model, not a string expression language. Each condition is a JSON object:
typescriptinterface RouteCondition {
    label: string;            // Display label on the edge (e.g., "Billing query")
    field: string;            // Output field to evaluate (e.g., "category", "sentiment")
    operator: RouteOperator;  // "equals" | "contains" | "startsWith" | "matches"
    value: string;            // Expected value
}
The UI presents this as a simple form: Field dropdown + Operator dropdown + Value input. For complex routing that can't be expressed as rules, users should use a dedicated "classifier" agent node instead.
At execution time, conditions are evaluated against the previous node's output metadata. A default edge (no condition) acts as the fallback.

public enum AIOrchestrationNodeType
{

    End,            // Exit point (one or more per graph)
    Agent,          // References an existing AIAgent by ID
    Function,       // M.E.AI AITool (registered tool, no LLM)
    Router,         // Conditional routing (rule-based switch)
    Aggregator,     // Merges concurrent results
    Manager         // Magentic pattern: manager agent that delegates dynamically

public sealed class AIOrchestrationNodeConfig
{

    public Guid? AgentId { get; set; }

    // Function node — references a registered AITool by name

    // Manager node (Magentic pattern)

public enum AIOrchestrationAggregationStrategy

    public required string Field { get; set; }          // Output field to evaluate

    public required string Value { get; set; }          // Expected value
    public required string TargetNodeId { get; set; }   // Edge target

public enum AIOrchestrationRouteOperator
{
    Equals,
    Contains,
    StartsWith,
    Matches         // Regex

### Workspace Structure

- **Collection view**: Table listing all orchestrations (Name, Status, Modified)
- **Editor view** with tabs:
  1. **Details** — Name, alias, description, active toggle
  2. **Workflow** — Full-width Rete.js canvas with `+` button and minimap
  3. **Availability** — Surface picker + scope rules (reuse agent surface picker component)
  4. **Info** — Created/modified metadata, version history

### Pattern Templates (Creation Modal)

When creating a new orchestration, a **dialog modal** presents options as visual cards:

- **Blank**: Empty canvas with just Start + End nodes
- **Sequential**: Start → Agent → Agent → End
- **Concurrent**: Start → [Agent, Agent, Agent] → Aggregator → End
- **Handoff**: Start → Agent (with handoff edges to other agents) → End
- **Group Chat**: Start → [Agent, Agent] (circular edges) → End
- **Magentic**: Start → Manager → [Agent, Agent] (bidirectional with Manager) → End

The selected template pre-populates the graph. After creation, the template choice is not stored — the graph is the source of truth.

## 5. Deploy Support (Umbraco.AI.Agent.Deploy)

Since orchestrations are part of the Agent package, deploy support is added to the existing `Umbraco.AI.Agent.Deploy` package — following the same pattern used for `AIAgent`.

### Deploy Components

**Artifact**: `AIOrchestrationArtifact` extends `DeployArtifactBase<GuidUdi>`
- Serializes: Alias, Name, Description, ProfileUdi (optional), SurfaceIds, Scope (JsonElement), Graph (JsonElement), IsActive
- Graph is serialized as a `JsonElement` — the entire node/edge structure transfers as JSON

**Service Connector**: `UmbracoAIOrchestrationServiceConnector`
- Extends `UmbracoAIProfileDependentEntityServiceConnectorBase<AIOrchestrationArtifact, AIOrchestration>` (since orchestrations have an optional ProfileId)
- UDI entity type: `"umbraco-ai-orchestration"`
- Process passes: `[3]` (same as agents — depends on profiles which deploy in pass 2)
- Pass 3: Create/update orchestration with all properties
- Pass 4: Resolve profile dependency

**Dependency tracking in `GetArtifactAsync`**:
- Profile dependency: `ArtifactDependencyMode.Match`
- Agent references within graph nodes: Each `Agent` node's `AgentId` is added as a dependency (`ArtifactDependencyMode.Match`) to ensure referenced agents deploy first

**Notification Handlers** (minimal — extend base classes):
- `AIOrchestrationSavedDeployRefresherNotificationAsyncHandler`
- `AIOrchestrationDeletedDeployRefresherNotificationAsyncHandler`

**Constants**:
- Add `Orchestration = "umbraco-ai-orchestration"` to `UmbracoAIAgentConstants.UdiEntityType`

6. Execution Engine
The AIOrchestrationExecutor translates the stored graph into MAF's WorkflowBuilder:
csharp// Pseudocode
public async Task<MsAIAgent> BuildWorkflowAgentAsync(AIOrchestration orchestration, CancellationToken ct)
{
    // 1. Create MAF agents for each Agent node
    var mafAgents = new Dictionary<string, MsAIAgent>();
    foreach (var node in orchestration.Graph.Nodes.Where(n => n.Type == AIOrchestrationNodeType.Agent))
    {
        var agentDef = await _agentService.GetAgentAsync(node.Config.AgentId!.Value, ct);
        mafAgents[node.Id] = await _agentFactory.CreateAgentAsync(agentDef, ct);
    }

    // 2. Resolve AITools for Function nodes
    var functions = new Dictionary<string, AITool>();
    foreach (var node in orchestration.Graph.Nodes.Where(n => n.Type == AIOrchestrationNodeType.Function))
    {
        functions[node.Id] = _toolRegistry.GetTool(node.Config.ToolName!);
    }

    // 3. Build workflow graph via WorkflowBuilder (edges, conditions, etc.)
    var builder = new WorkflowBuilder(/* entry agent */);
    foreach (var edge in orchestration.Graph.Edges)
    {
        // Wire direct edges, conditional edges, etc.
    }

    // 4. Return workflow — can be used as an agent via MAF composability
    return builder.Build().AsAgent();
}
